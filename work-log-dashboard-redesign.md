## 現状の課題

自分のダッシュボード（index.html）を振り返って気づいたこと:

1. **全パネルがフラットに並んだ分析画面**。一回見たら終わる
2. **平日1分で見るべきものと週末15分で見るべきものが混在**。dashboard-checklist.mdでワークフローは設計済みなのに、画面がそれに対応していない
3. **散布図が機能していない**。自己紹介記事（PV705）が外れ値で右上に飛び、他100本が左下に団子状態。カテゴリ色分けも重なって読めない
4. **スパークラインを「ふーん」で流している**。データは見えるけど、何を判断すればいいか分からない
5. **wrapped.htmlとの中途半端な分離**。dashboardは「分析用だけどキャラがいない」、wrappedは「キャラはいるけど判断に使えない」

## 思考の流れ

### 「チェックリストの手順 = 画面の順序にすればいい」

dashboard-checklist.mdは既に完成している:
- 平日: 状態メモをコピー→ミニログPWAに貼るだけ。1分で終わらせる
- 週末: カテゴリバランス→η序列→散布図→スパークライン→アクション。15分

このチェックリストの順番通りにタブとセクションを並べれば、「何をどの順で見るか」に迷わない。

### 「ツール間の役割重複を避ける」

全ツールを洗い出した:
- おはヨミ（推しクリエイターの記事読み）
- おへんじ帖（未返信コメント管理）
- ミニログPWA（判断ログ→Box）
- 積み上げノート（フォロワー推移。他ユーザーにも公開中）
- note-tag-collector（タグ共起ネットワーク）

ダッシュボードに全部詰め込むのではなく、各ツールの仕事を尊重する。
ただしフォロワー推移グラフはダッシュボードにも欲しい（積み上げノートは「公開PWA」としての別役割）。

### 「散布図の問題は見せ方の問題」

散布図自体が悪いのではなく、外れ値でスケールが壊れている。
→ 散布図はDeep Diveに移して、Weeklyには4象限テーブル（記事を分類して並べる）を置く方が判断に使える。

### 「キャラ演出はKITAさんにはできない差別化」

KITAさんが「ワイ語」でコピーを統一したように、自分は7人のカノジョキャラで統一できる。
wrappedのキャラ演出（girl-lines.jsのナビゲーター+データ連動セリフ）をダッシュボードに統合する。
→ 「キャラがデータを解釈してくれるダッシュボード」はおはようカノジョのコンテンツそのもの。

### 「weekly_report.pyは使ってなかった」

テキスト出力だと見ない。ロジックは有用（カテゴリバランス判定、η序列チェック、A+B伸び傾向）なので、ダッシュボードのWeeklyタブに視覚化して組み込む。

### 「スパークラインは見るだけで終わっている」

データを見せるだけでなく、キャラが解釈を言語化してくれれば判断につながる。
→ Weeklyタブに「今週のハイライト」セクション:
  - 最もPV伸びた記事 → 月子が分析コメント
  - 最もスキ率高い記事 → 陽が褒める
  - 伸びなかった記事 → 凛華がツッコミ

## Phase 2 実装メモ（2026-03-20）

### 設計で最も時間を使ったこと

**タブ構成の3回の変更**。最初は Daily/Weekly/Monthly/Deep Dive の4タブで設計したが、実装しながら何度も見直した。

1. **最初**: Weekly に「フォロワー推移」「スキしてくれた人」「記事振り返り」「カテゴリバランス」「アクション」を全部詰め込んだ
2. **気づき**: Weeklyが「コミュニティ活動」と「記事戦略」の2つの目的を兼ねていた。スキ返しは毎日やるべきで、週末まで溜めるのは変
3. **最終形**: Activity タブを分離。Daily=ミニログの材料、Activity=スキ活の起点、Weekly=記事戦略

**「Dailyタブで何を見てどうアクションするか」の議論**。Hidden Gemsを入れたが「見てどうする？」と問われて外した。スキ増ランキングも「アクション不要では？」となったが、ミニログの判断材料になると気づいて残した。**データを見せるだけでなく、ワークフローの中での役割を常に問う**必要があった。

### UIデザインの試行錯誤

**チャット形式 → ナビ+カード形式への回帰**。最初はLINEのようなチャット形式（吹き出し+アバター）で実装したが、データカードが吹き出しの中に入ると見づらかった。wrappedと同じ「ナビゲーター（キャラ+セリフ）がセクションの上に立つ」形式に戻した。チャット形式は「キャラとの対話感」は出るが、データの視認性を犠牲にする。

**配色の迷走**。Dailyタブのクリーム背景を試したが「内容との違和感」で即戻し。Weeklyも明るい背景を試したが「目が疲れる」で戻し。最終的にダークテーマ統一。**ダッシュボードは夜に見るものだから、ダークが正解だった。** 吹き出しの明るい色も試したが、半透明の三角部分が本体との境界で目立つ問題が発生。不透明色（#242433）で解決。

**キャラのパーソナルカラー**をアイコン枠とキャラ名に適用してみたが、配色が増えすぎてダッシュボードが散らかった。シアン統一に戻した。**差別化はセリフの内容で出すべきで、色で出すと情報過多になる。**

### セリフ設計の発見

**セリフには「データの主語」を含めないとわからない**。「まあ、悪くないわね。」だけだと何について言っているか不明。「フォロワー+7。...まあ、悪くないわね。」にすることで、ラベルが不要になった。セリフ自体がラベルの役割を兼ねる。

**セリフの閾値設計**。girl-lines.jsの`GIRL_DYNAMIC`に、フォロワー差分（+10/+5/+1/0/-2/-3以下）やスキ増（+10/+5/+1/0）の閾値ごとにキャラセリフを定義。wrappedの仕組みをDaily/Weekly用に拡張。Weeklyでは7人のキャラが担当セクションを語る形に。

### スキしてくれた人の分類

**新規/復帰/常連/たまに/離脱危機の5分類**を実装。目的は「スキ活・コメント活動の優先順位がわかること」。

- 新規: 今週初めてスキしてくれた人 → 即スキ返し
- 復帰: 直近4週来てなかったが今週復帰 → 再接続
- 常連: 直近4週中3週以上 → 関係維持
- たまに: 不定期 → 余裕があれば
- 離脱危機: 以前頻繁だったが直近4週来ていない → 読みに行く

**プロフィール画像はCFプロキシ経由で非同期取得**。`?id=`パターンを使い、KV書き込みが発生しない安全な方式を選択。将来はCSVにプロフィール画像URLを保存してAPI不要にする設計。`getProfileImageUrl()`関数で抽象化し、切り替え時に呼び出し元は変更不要。

### Dailyタブの最終形

「ミニログを書くための材料を揃える場所」として確定。

- 日付ナビ+Copyボタン（タブ上部に固定）
- 状態メモ（矢印+コメント）
- フォロワー前日比
- 今日の記事（あれば。平均PV/スキとの比較付き）
- スキ増ベスト3（前日比。累計スキ数も表示）
- 観測は、続く。

**全セクションをその日のキャラ1人がナビゲート**。Activityは複数キャラ、Weeklyも複数キャラ。
